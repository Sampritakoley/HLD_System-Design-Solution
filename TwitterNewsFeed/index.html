01 / Understanding the problem

What is this system, really?

Twitter is a microblogging and social networking platform. Users write short messages called tweets (up to 280 characters), follow other users, and see a feed of tweets from people they follow. Think of it as a global public notice board — anyone can post, anyone can read, and content spreads virally through retweets and likes.

The core challenge: unlike Instagram (mostly photos), Twitter is pure text at massive velocity. News breaks on Twitter. Trends emerge in minutes. A single tweet from a president or celebrity can trigger billions of reads within seconds. Designing for this unpredictable, bursty, read-heavy traffic is what makes this problem uniquely hard.

Core features in scope

Tweet — post a message

User writes up to 280 characters, optionally attaches media, hits post. Must be stored durably, indexed for search, and distributed to all followers.

News Feed — home timeline

The most complex feature — must aggregate tweets from all followed users, rank them, and serve in under 200ms. The hardest engineering problem in the system.

Follow — the social graph

Follow/unfollow relationships that power everything. Who follows who determines whose tweets appear in whose feed.

Search & Trending — find content

Real-time search across all tweets and hashtags. Trending topics updated every few minutes, requiring near-real-time indexing at massive scale.

What makes this hard at scale

  • Extreme read:write ratio — 100:1. For every tweet posted, it gets read ~100+ times. The system must be optimized for reads above everything.
  • The celebrity problem — worse than Instagram. Elon Musk has 150M followers. One tweet triggers 150M feed updates. The fan-out math is catastrophic at scale.
  • Viral spikes are unpredictable. A single breaking news tweet can go from 0 to 10M reads in 60 seconds. The system must absorb this without pre-warning.
  • Real-time expectation. Users expect to see a tweet within seconds of posting. Unlike Instagram where a few seconds delay is fine, Twitter users notice latency of even 5–10 seconds.
02 / Requirements

Functional vs. non-functional requirements

Non-functional requirements are what actually drive architecture decisions — they're listed first for a reason.

Functional

  • Post tweets (text, images, videos)
  • Follow / unfollow users
  • View home timeline (news feed)
  • Like, retweet, reply
  • Search tweets and hashtags
  • View trending topics
  • Real-time notifications
  • Direct messages

Non-functional

  • 300M DAU, 500M tweets/day
  • Read:Write ratio = 100:1
  • Feed load < 200ms (p99)
  • Tweet post < 500ms
  • 99.99% availability
  • Eventual consistency OK for feed
  • Strong consistency for tweets (don't lose data)
  • Global scale, multi-region

Clarifying questions worth asking first

Are we supporting media (images/video) in tweets, or text only? Changes storage architecture significantly.
Is the feed chronological or ranked by an ML algorithm?
Do we need to handle verified/celebrity accounts differently in feed generation?
What's the tweet character limit — 280 or supporting long-form posts?
Do we need real-time push notifications, or is polling acceptable?
Is search full-text across all tweets, or just hashtags and usernames?
What consistency model for the feed — can followers see tweets a few seconds late?
What geographic distribution — global or single region to start?

Component rationale

CDN
Media files and static assets. At 200 GB/sec download peak, you cannot serve this from origin. CDN caches at 200+ edge locations globally. A user in Delhi gets media from a node in Delhi, not San Francisco.
API Gateway / Load Balancer
Single entry point. Handles auth (JWT validation), rate limiting (Twitter limits 300 requests per 15 min per user), routing to the correct microservice, and SSL termination.
Tweet Service
Validates tweet content, stores to Cassandra, publishes event to Kafka. Must be stateless and horizontally scalable. Returns success to user in <500ms — all post-write processing happens async.
Kafka (message queue)
Backbone of the system. Every tweet event is published here. Fan-out workers, notification workers, search indexers, and analytics all consume from Kafka independently. Decouples tweet posting from all downstream effects.
Fan-out Workers
Consume from Kafka. For each new tweet, fetch the author's follower list and push tweet ID to each follower's Redis timeline. This is where the celebrity problem must be solved (see deep dive).
Redis (timeline cache)
Each user's home timeline is stored as a sorted set of tweet IDs in Redis. Reading 20 tweet IDs takes under 1ms. Without Redis, every feed load would require an expensive database join. 600K reads/sec at <200ms is only possible with in-memory storage.
Cassandra (tweet store)
High write throughput (17K/sec), time-series access patterns, horizontal scale. LSM-tree writes are sequential so writes are extremely fast. Tweets are immutable after posting — perfect for Cassandra's append-only model.
Elasticsearch
Full-text search across all tweets, hashtag indexing, trending topic computation. Updated in near-real-time via Kafka consumer. Too expensive to run on Cassandra or Postgres at this scale.
PostgreSQL (user/follow store)
Follow relationships need ACID — you can't half-follow someone. Also stores user profiles, settings, and verified status. Moderate write volume, strong consistency required.

End-to-end data flow — "user posts a tweet"

  1. User types tweet, hits post. App sends POST /tweets to API Gateway.
  2. API Gateway validates JWT token, checks rate limit.
  3. Tweet Service validates content (length, media), assigns a Snowflake ID, stores tweet to Cassandra, returns "tweet posted!" to user in <500ms.
  4. Tweet Service publishes to Kafka: {tweet_id, user_id, timestamp, hashtags}
  5. Fan-out Worker (async): fetches author's follower list from Postgres, pushes tweet ID to each follower's Redis sorted set. Trims timeline to last 800 tweets.
  6. Search Indexer Worker (async): reads from Kafka, indexes tweet content in Elasticsearch.
  7. Notification Worker (async): if tweet mentions @someone, sends push notification.
  8. When follower opens Twitter: Feed Service reads ZRANGE timeline:user_B 0 19 from Redis (<1ms), batch-fetches tweet content from Cassandra by IDs, returns assembled feed in <200ms.
05 / Database design

Storage layer

Different data, different access patterns, different databases — not one engine forced to do everything.

DataDatabaseWhy
TweetsCassandra17K writes/sec, immutable, time-series, horizontal scale
Users, followsPostgreSQLACID, relational, moderate volume, strong consistency

Tweets table (Cassandra)

CREATE TABLE tweets (
  user_id       UUID,
  tweet_id      BIGINT,      -- Snowflake ID (encodes timestamp)
  content       TEXT,
  media_url     TEXT,
  like_count    INT,
  retweet_count INT,
  PRIMARY KEY (user_id, tweet_id)
) WITH CLUSTERING ORDER BY (tweet_id DESC);

-- Query: latest 20 tweets by user_123
SELECT * FROM tweets WHERE user_id='user_123' LIMIT 20;

-- Partition key = user_id → all tweets by one user on same node
-- Clustering key = tweet_id → sorted by time automatically (Snowflake)

Users & follows tables (PostgreSQL)

CREATE TABLE users (
  user_id        UUID PRIMARY KEY,
  username       VARCHAR(15) UNIQUE,
  email          VARCHAR(255) UNIQUE,
  follower_count INT DEFAULT 0,   -- denormalized counter
  is_verified    BOOLEAN DEFAULT false,
  created_at     TIMESTAMPTZ
);

CREATE TABLE follows (
  follower_id UUID REFERENCES users(user_id),
  followee_id UUID REFERENCES users(user_id),
  created_at  TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (follower_id, followee_id)
);
CREATE INDEX idx_follows_followee ON follows(followee_id);
-- Index 1: "who does user A follow?"   → query by follower_id
-- Index 2: "who follows user A?"        → query by followee_id

Redis timeline structure

Key:   timeline:{user_id}
Type:  Sorted Set (ZSET)
Score: tweet timestamp (Unix ms)   ← enables time-ordering
Value: tweet_id

ZADD timeline:user_B 1703001234000 tweet_id_xyz
ZRANGE timeline:user_B 0 19 WITHSCORES   → top 20 tweets
ZREMRANGEBYRANK timeline:user_B 0 -801   → keep only 800

Sharding strategy

  • Tweets — shard by user_id; all tweets by the same user on one Cassandra node. The dominant query "get tweets by user X" hits exactly one node.
  • Follows — shard by follower_id; fan-out workers need "who does user X follow?" — shard on the key queried most during feed generation.
  • Timelines (Redis) — partition by user_id across multiple Redis clusters using consistent hashing.
  • Consistent hashing for routing — adding new nodes only moves 1/N of data. No full reshuffle. Critical at 37PB scale where migrations are multi-day operations.
06 / Deep dives

Deep dive on 3 critical components

For each: the problem, the naive solution, why it fails, and the real solution used at scale. Volunteer these problems before the interviewer asks — it signals senior-level thinking.

01 Feed generation — the fan-out problem

The problem: When Elon Musk tweets, 150 million followers need to see it in their feed within seconds. How do you push one tweet to 150M users without melting your servers?

Naive solution — pull model (fan-in at read time)
When user opens Twitter:
  SELECT tweets.* FROM tweets
  JOIN follows ON tweets.user_id = follows.followee_id
  WHERE follows.follower_id = 'me'
  ORDER BY tweet_id DESC LIMIT 20;

Why it fails: JOIN across billions of rows at 600K reads/sec = database destroyed immediately. No latency target survives this.

Real solution — hybrid push/pull model
Regular users (< 1M followers) → PUSH model:
  New tweet → Kafka → Fan-out Worker
  → ZADD timeline:each_follower tweet_id score
  → Feed pre-built in Redis before anyone reads

Celebrities (> 1M followers) → PULL model:
  Skip fan-out entirely.
  Store tweet in celebrity_tweets table.
  At READ time, merge:
    1. Redis timeline (regular accounts user follows)
    2. Fresh fetch of celebrity tweets user follows
    3. Merge by timestamp, rank, return top 20
    4. Cache merged result for 2 minutes

Why 2 min cache not 5? Twitter moves faster than Instagram.
News breaks in seconds. 2 min is the sweet spot for freshness vs cost.
The key insight: 99% of users use the push model — instant Redis read under 1ms. The 1% who are celebrities skip fan-out entirely. At read time, merge both lists, cache briefly. Feed stays fresh, servers stay alive.
02 Tweet ID generation — the Snowflake algorithm

The problem: We generate 17,000 tweet IDs per second across hundreds of servers. Each ID must be globally unique, sortable by time, and generated without coordination between servers.

Naive solution — auto-increment integer

Why it fails: In a distributed system, auto-increment requires a central coordinator. All 17,000 writes/sec must route through one server to get the next ID. That server becomes a single point of failure and a bottleneck. If it goes down, tweeting stops globally.

Real solution — Twitter Snowflake ID
Snowflake ID = 64-bit integer composed of:

┌─────────────┬──────────┬────────────┬──────────────┐
│ 1 bit sign  │ 41 bits  │ 10 bits    │ 12 bits      │
│ (always 0)  │timestamp │ machine ID │ sequence num │
└─────────────┴──────────┴────────────┴──────────────┘

Example:
  timestamp  = ms since Twitter epoch (Nov 4, 2010)
  machine_id = unique ID of the server generating this
  sequence   = counter that resets every millisecond

Result:
  → Each machine generates IDs independently (no coordination)
  → IDs are sortable by time — higher ID = newer tweet
  → 4096 unique IDs per millisecond per machine
  → No central coordinator = no single point of failure
This is why Twitter IDs look like "1741234567890123456" — that huge number is a 64-bit integer, not random. The first ~41 bits encode time, which is why sorting by ID gives you chronological order for free, at zero query cost.
03 Trending topics — real-time computation

The problem: Twitter shows trending hashtags and topics updated every few minutes. At 500M tweets/day, how do you count hashtag frequency in real-time across the entire platform?

Naive solution — batch count
Every 5 minutes, run:
  SELECT hashtag, COUNT(*) FROM tweets
  WHERE created_at > NOW() - INTERVAL '1 hour'
  GROUP BY hashtag ORDER BY count DESC LIMIT 10;

Why it fails: Full table scan across billions of tweets every 5 minutes = crushing DB load. Also not real-time — by the time you compute it, the trend may have already peaked and died.

Real solution — streaming count with sliding window
Step 1: Kafka stream
  Every tweet published to Kafka includes extracted hashtags

Step 2: Stream processing (Apache Flink)
  Consumer reads from Kafka continuously
  Maintains in-memory count per hashtag
  Uses a 1-hour sliding window:
    every 1 min → add counts from last 60 seconds
    every 1 min → subtract counts from 61+ minutes ago

Step 3: Store top-K in Redis
  ZADD trending:global hashtag count
  Only keep top 1000 hashtags in Redis ZSET

Step 4: Serve trending API
  ZRANGE trending:global 0 9 WITHSCORES → top 10 trends
  Refresh every 30 seconds → users see near-live trends

Why Flink not Spark? Flink is true streaming (per-event).
Spark Streaming is micro-batch (every few seconds).
For trending, per-event updates feel more real-time.
07 / Tradeoffs & alternatives

What we did NOT choose and why

Rejected alternatives

AlternativeWhy rejected
MySQL for tweetsCan't handle 17K writes/sec. B-tree index updates are too slow. Cassandra's LSM tree and sequential writes win at this scale.
Pure push model for everyoneElon Musk (150M followers) posting = 150M Redis writes. At 100K writes/sec that takes 25 minutes. Hybrid model is mandatory.
Pure pull model for everyone600K feed reads/sec each doing a JOIN = database instantly overwhelmed. Pre-computed Redis timelines are not optional.
Auto-increment tweet IDsCentral coordinator bottleneck. Single point of failure. Snowflake IDs give you distributed, time-sortable IDs with no coordination.
Batch trending computationTrend peaks and dies in minutes. A 5-minute batch gives you yesterday's news. Flink streaming gives near-real-time results.
WebSockets for all APIsStateful connections are hard to scale horizontally. Reserve WebSockets/SSE for real-time notifications only. REST for everything else.
Single global Redis clusterA single Redis cluster has memory limits. Partition timelines by user_id across multiple Redis clusters using consistent hashing.

What breaks at 10× scale and how we fix it

Problem — Redis memory exhaustion

300M users × 800 tweet IDs × 8 bytes = ~2TB RAM. At 10×, that's 20TB — too expensive.

Fix

Only keep timelines for users active in the last 7 days. Recompute on first login for inactive users. Compress tweet IDs using delta encoding.

Problem — Kafka consumer lag during viral events

A viral tweet triggers 1B fan-out writes. Kafka consumer groups fall behind and timelines go stale.

Fix

Auto-scaling consumer groups. Priority lanes in Kafka — verified users get higher-priority partitions. Celebrity tweets bypass fan-out entirely (already handled by pull model).

Problem — Cassandra hot partition (viral tweets)

A viral tweet gets 10M reads in 60 seconds — all hitting the same Cassandra partition (the user_id of the tweeter).

Fix

Add read replicas per hot partition. Cache individual tweet content in Redis with short TTL (60 seconds). CDN-cache tweet content for media-heavy tweets.

Problem — Elasticsearch indexing lag

At 170K tweets/sec at 10× scale, Elasticsearch cannot index fast enough for real-time search.

Fix

Tiered search — last 7 days in hot Elasticsearch nodes (fast SSD), older content in warm nodes (HDD). Use Kafka consumer groups with a separate indexing pipeline that can scale independently.

08 / Summary

Cheat sheet

The ten things worth remembering before walking into the room.

Key numbers

300M DAU · 17K write QPS · 600K read QPS · 150GB text/day · 20TB media/day · Read:Write 100:1

Key databases

Cassandra (tweets) · PostgreSQL (users/follows) · Redis (timelines/counters) · Elasticsearch (search/trending) · S3 (media)

Feed strategy

Push to Redis for regular users (<1M followers). Pull at read time for celebrities. Merge + cache 2 min at read time.

Tweet ID

Snowflake ID = 41-bit timestamp + 10-bit machine ID + 12-bit sequence. Distributed, no coordinator, time-sortable.

Trending topics

Kafka → Flink streaming → 1-hour sliding window count → top-K in Redis ZSET → refresh every 30 seconds.

Key tradeoffs

Eventual consistency on feed (acceptable). Strong consistency on tweets (mandatory). Push vs pull for fan-out. Real-time vs batch for trending.

Hardest part

Celebrity fan-out. 150M Redis writes for one tweet = 25 min lag. Hybrid pull model for celebrities is the key insight.

Kafka is the backbone

Every tweet event flows through Kafka. Fan-out, search indexing, notifications, trending, analytics — all independent consumers.

Sharding keys

Tweets → user_id · Follows → follower_id · Timelines → user_id across Redis clusters · Consistent hashing for all.
