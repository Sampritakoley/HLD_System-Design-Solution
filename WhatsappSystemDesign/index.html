01 / Understanding the problem

What is this system, really?

Imagine two people on opposite sides of the planet, each holding a phone that may lose signal at any moment. WhatsApp is the system that lets one of them type a message and have it appear on the other's screen — reliably, in order, within a second — even if the recipient's phone was off when it was sent, and even though the company running the servers is never able to read the message itself.

It is not just a database of messages. It's a real-time, low-latency delivery network built on top of millions of long-lived connections, layered with a durability guarantee (nothing gets lost), an ordering guarantee (messages arrive in the order they were sent), and an end-to-end encryption guarantee (only the sender and recipient devices can ever decrypt the content). Think of it as a postal service where every letter is sealed in a box only the recipient has the key to, delivered instantly if they're home, and held safely at the door if they're not.

Core features in scope

1:1 messaging

Send text, photos, videos, voice notes, and documents between two users with delivery and read receipts.

Group messaging

One message fans out to every member of a group, each with their own encrypted copy and delivery state.

Presence & status

Online / last-seen / typing indicators — low-value data individually, but extremely high write volume in aggregate.

Media sharing

Photos, videos, and voice notes — encrypted, uploaded once, referenced by pointer, and delivered via CDN.

What makes this hard at scale

  • Millions of concurrent persistent connections. Unlike HTTP request/response, a chat app needs an open socket per active device so the server can push messages instantly — that's a fundamentally different scaling problem than stateless web traffic.
  • Devices go offline constantly. Phones lose signal, apps get killed, batteries die — the system must hold undelivered messages safely and flush them the instant a device reconnects.
  • Ordering must survive distributed writes. Messages in the same conversation must render in the order they were sent, even if they were processed by different servers.
  • End-to-end encryption means the server can't help itself. The server can't index message content, can't deduplicate on content, and can't debug delivery issues by reading payloads — it's routing ciphertext blindly.
  • Group fan-out multiplies everything. A single message to a 250-person group means encrypting, storing, and delivering roughly 250 separate copies.
02 / Requirements

Functional vs. non-functional requirements

Non-functional requirements are what actually drive architecture decisions here — a chat system lives or dies on latency and durability, not feature count.

Functional

  • Send/receive 1:1 text, image, video, voice, document messages
  • Create and message groups (up to ~1000 members)
  • Delivery receipts (sent / delivered / read)
  • Online/last-seen presence, typing indicators
  • Multi-device sync (same account, phone + linked devices)
  • Offline message delivery once a device reconnects

Non-functional

  • Billions of users, tens of billions of messages/day
  • End-to-end message delivery latency < 1s (p99, online recipient)
  • End-to-end encryption — server never sees plaintext
  • Zero message loss once "sent" is acknowledged — durability first
  • Prioritize availability over strict consistency (AP)
  • Message ordering preserved per conversation

Clarifying questions worth asking first

Do we need to support group chats in this scope, or just 1:1 messaging?
Is end-to-end encryption a hard requirement, or is encryption-in-transit (TLS) to the server sufficient?
Do we need multi-device support — same account logged into a phone and a linked desktop client simultaneously?
How long should undelivered messages be retained server-side before we can safely drop them — days, weeks?
Is media (photos/video/voice notes) in scope, or text-only messaging?
Do we need presence (online/last-seen) and typing indicators, given how write-heavy they are?
Single region or globally distributed users?
Do we need message search or server-side content indexing — which would conflict with E2E encryption?

Assumptions

DAU                        = 500M users
Messages sent/user/day     = 40
Concurrent-online fraction = ~30% of DAU at peak
Avg text message size      = ~100 bytes (+ metadata overhead ~100 bytes)
Media messages             = 10% of all messages
Avg media size             = ~150KB (mix of photo/voice-note/video-thumbnail)

QPS

Messages/day:
  500M × 40                    = 20,000,000,000 (20B) messages/day
  Average QPS: 20B / 86,400    = ~231,000/sec
  Peak (3×)                    = ~700,000/sec

This is a WRITE-heavy real-time system, unlike a feed — every message is
both a "write" (persist) and effectively a "read" (immediate push to recipient).

Concurrent connections

Peak concurrent online users = 500M × 30%   = ~150,000,000 connections

If one gateway server holds ~50,000 persistent connections:
  Servers needed = 150M / 50K = ~3,000 gateway servers

This single number is the reason connection management gets its own
service layer, separate from request-handling app servers.

Storage

Text metadata per day:
  20B × 90% × 200 bytes (msg + metadata) = 3.6 TB/day
  Over 1 year: 3.6TB × 365               = ~1.3 PB/year

Media per day:
  20B × 10% × 150KB                       = ~300 TB/day

Total storage/day ≈ ~304 TB/day

Bandwidth

Text message round-trip (send + push to recipient) ≈ 400 bytes total
Peak text bandwidth  = 700,000/sec × 400 bytes ≈ 280 MB/sec (~2.2 Gbps)

Conclusion: text is cheap; media dominates both storage and bandwidth —
media must be offloaded to Blob storage + CDN, never routed through the
same path as real-time message delivery.
04 / High-level design

Architecture

Built top-down from the client. Every box exists because of a number from Step 3 — especially the 150M concurrent connections and the E2E encryption requirement.

Component rationale

Connection Gateway Layer
Holds the actual persistent WebSocket/TCP connection per online device. Kept as its own stateless-per-connection layer, separate from business logic, so it can be scaled purely by connection count (~50K/node) rather than request throughput.
Connection Registry (Redis)
A distributed map of user_id/device_id → which specific gateway node holds their live socket. Without this, no other service knows where to push a message for delivery to an online user.
Chat / Messaging Service
Core business logic — validates the message envelope, assigns a monotonic per-conversation sequence number, decides the delivery path (push if online, mailbox if offline), and orchestrates the receipt lifecycle (sent → delivered → read).
Message Store (sharded Cassandra — "mailbox" model)
Every message is durably appended to a per-recipient mailbox before anything else happens — this is what makes "zero message loss" possible even if a push attempt fails. Sharded by user_id/conversation_id for even write distribution.
Presence Service (Redis)
Tracks online/last-seen/typing state — extremely high write volume, low durability requirement, so an in-memory store with short TTLs fits better than a durable DB.
Group Fan-out Service + Kafka
For group messages, decouples "one message sent" from "N per-recipient delivery jobs" — Kafka absorbs the burst so a 1000-member group post doesn't block the sender's request path.
Media Service + Blob Storage + CDN
Media is encrypted client-side, uploaded once to Blob storage, and referenced by an opaque pointer + decryption key sent inside the encrypted message itself — the server stores ciphertext bytes it cannot interpret.
Push Notification Service (APNs / FCM)
When a recipient device has no live connection, this is what wakes the OS-level app process so it can reconnect and drain its mailbox — the actual message content is never put in the push payload itself.

End-to-end data flow — "User A sends a message to User B (1:1)"

  1. Client A encrypts the message client-side using the Signal Protocol (per-recipient-device session key), producing ciphertext the server cannot read.
  2. Client A sends the ciphertext over its already-open persistent connection to whichever Gateway node it's attached to.
  3. Gateway forwards the envelope to the Chat Service, which assigns a monotonic sequence number for that conversation.
  4. Chat Service durably appends the ciphertext to User B's mailbox in the Message Store — this happens before anything else, guaranteeing durability.
  5. Chat Service checks the Connection Registry: is User B online? If yes, it routes the message directly to the Gateway node holding B's socket for near-instant push delivery.
  6. If User B is offline, the message stays queued in the mailbox and a silent push notification is sent via APNs/FCM to wake B's device.
  7. Once B's client receives and acknowledges the message, a "delivered" receipt flows back through the same pipe to A; a "read" receipt follows once B's client marks it read.

End-to-end data flow — "Group message fan-out"

  1. Sender's client encrypts the message separately per recipient device (E2E encryption means there's no single shared ciphertext — one payload per device in the group).
  2. These per-recipient jobs are pushed onto Kafka rather than processed synchronously in the request path.
  3. The Group Fan-out Service consumes each job and repeats the same per-recipient mailbox-write + online-push-or-offline-notify flow as the 1:1 case.
  4. Delivery/read receipts are tracked per member, and the group UI aggregates them (e.g. "read by 12 of 40").
05 / Database design

Storage layer

Multiple specialized stores, not one database — connection state, durable messages, and media each have wildly different access patterns and durability needs.

DataDatabaseWhy
Message mailboxesSharded CassandraHigh append-only write throughput, simple key-based lookups by user_id, horizontally scalable
Connection registryRedisNeeds sub-millisecond lookups on every message routed to an online user, low durability requirement
Presence / last-seenRedis (TTL-based)Extremely high write volume, short-lived data, no need for durable storage
Media (photos/video/voice)Blob/Object storage + CDNLarge encrypted binary blobs, immutable once uploaded, best served via CDN edge nodes
Group membership/metadataSharded relational DBSmall, structured, relatively low write volume, needs consistent reads

Message mailbox table

CREATE TABLE mailbox (
  recipient_id     BIGINT,
  conversation_id  BIGINT,
  sequence_no      BIGINT,      -- monotonic per conversation, preserves ordering
  sender_id        BIGINT,
  ciphertext       BLOB,        -- opaque to the server
  media_ref        VARCHAR,     -- optional pointer to Blob storage
  created_at       TIMESTAMP,
  delivery_state   ENUM (sent, delivered, read),
  PRIMARY KEY (recipient_id, conversation_id, sequence_no)
);

-- Sharded by recipient_id: co-locates everything one user needs to
-- drain on reconnect into a single-shard range scan.

Connection registry (Redis)

key:   conn:{user_id}:{device_id}
value: gateway_node_id, last_heartbeat_ts

Presence key (separate, shorter TTL):
  key: presence:{user_id} → {online | last_seen_ts}, TTL ~30s,
  refreshed by heartbeat from the client's open connection.

Sharding strategy

  • Mailboxes — shard by recipient_id; every read a device does on reconnect ("give me everything since sequence N") is a single-shard range scan.
  • Connection registry — sharded/partitioned by user_id hash across a Redis Cluster so a lookup during message routing is a single-node operation.
  • Group membership — sharded by group_id, since group metadata is read together far more often than per-user.
  • Consistent hashing across all shards so gateway nodes and Redis nodes can be added/removed without a full reshuffle — only neighboring partitions are affected, minimizing forced reconnects.
06 / Deep dives

Deep dive on 3 critical components

For each: the problem, the naive solution, why it fails, and the real solution used at scale. Volunteer these problems before the interviewer asks — it signals senior-level thinking.

01 Persistent connection management at scale

The problem: A chat app needs an open, addressable connection per online device so messages can be pushed instantly. At 150M concurrent connections, how do you even hold that many sockets open, let alone route a message to the exact right one?

Naive solution — stateless load balancer, no session affinity

Route each new connection to any available gateway server round-robin, with no record of who's connected where.

Why it fails: When the Chat Service needs to push a message to User B, nothing tells it which of the ~3,000 gateway nodes is holding B's socket right now — you'd have to broadcast to every node, which doesn't scale past a handful of servers.

Real solution — sticky connections + distributed connection registry
On connect:
  Gateway node accepts the socket, registers
  conn:{user_id}:{device_id} → this_node_id in Redis,
  with a heartbeat refresh every ~15-30s.

On message routing:
  Chat Service looks up the registry for the recipient's
  node_id, then forwards the message directly to that
  gateway instance over an internal RPC — O(1) lookup,
  no broadcast.

On disconnect / node failure:
  Registry entry expires via TTL if heartbeats stop;
  client reconnects, often routed near its old node via
  consistent hashing to reduce registry churn.

Gateway nodes use async, event-driven I/O (epoll-style) rather than
one OS thread per connection, since one thread per socket doesn't
scale to 50K connections per node.
This turns an "unaddressable ocean of sockets" problem into an O(1) lookup problem — the registry is the single most load-bearing piece of the whole real-time delivery path.
02 Guaranteed delivery & ordering (offline devices, multi-device sync)

The problem: Phones go offline constantly. A message sent while the recipient's phone is dead must not be lost, must eventually arrive, and must render in the same order it was sent — even across multiple devices on the same account.

Naive solution — fire-and-forget over the socket

Push the message directly to the recipient's connection; if the socket isn't open, drop it.

Why it fails: Any message sent while the recipient is offline is silently lost forever — completely unacceptable for a messaging product, and there's no way to recover ordering if multiple servers handle different messages in a conversation concurrently.

Real solution — durable mailbox + monotonic sequencing + ACK state machine
Before any push attempt:
  Message is durably appended to the recipient's mailbox
  with a sequence_no monotonically increasing per
  conversation — assigned by a single sequencer per
  conversation shard, so ordering is never ambiguous
  even under concurrent writers.

Delivery attempt:
  If recipient online → push immediately, mark 'delivered'
  on client ACK.
  If offline → leave in mailbox, send a silent push
  notification; on reconnect, client requests
  "everything since my last seen sequence_no" and the
  server streams the backlog in order.

State machine per message: sent → delivered → read,
each transition acknowledged explicitly by the
recipient's client, never assumed by the server.

Multi-device: each linked device tracks its own
last-synced sequence_no per conversation, so a phone
and a desktop client can independently catch up without
duplicating or skipping messages.
The core trick is doing the durable write *before* attempting delivery — delivery is best-effort and retryable, but the write is the actual guarantee.
03 End-to-end encryption & key management

The problem: The server must relay and store messages without ever being able to read their content — even if the servers themselves are compromised, and even for group chats with hundreds of members.

Naive solution — encrypt in transit only (TLS to the server)

Encrypt the connection between client and server, decrypt on arrival, store and forward in plaintext.

Why it fails: The server (and anyone who compromises it, or is compelled to hand over data) can read every message — this fails the core privacy guarantee the whole system is built around.

Real solution — Signal Protocol (X3DH handshake + Double Ratchet)
Key setup (once per device):
  Each device publishes an identity key, a signed prekey,
  and a batch of one-time prekeys to a key server.

Session start (X3DH handshake):
  Sender fetches recipient's public prekey bundle and
  derives a shared secret locally — the server only ever
  sees public keys, never the derived secret.

Per-message ratcheting (Double Ratchet):
  Every message uses a new derived key, ratcheted forward
  from the last one. Compromising one message's key does
  not expose past or future messages — a property called
  forward secrecy.

Group encryption (sender keys):
  Rather than pairwise-encrypting to every member
  (quadratic cost), the sender distributes a single
  "sender key" per group member once; subsequent group
  messages are encrypted once and fanned out, each
  member decrypting with the sender key they hold.

The server's job in this entire flow is limited to storing and
routing opaque ciphertext blobs — it structurally cannot
decrypt them, even under compulsion.
This is why the Message Store table above only ever holds a ciphertext BLOB column — the server-side schema itself reflects the privacy guarantee.
07 / Tradeoffs & alternatives

What we did NOT choose and why

Rejected alternatives

AlternativeWhy rejected
Long-polling / short-polling instead of persistent connectionsWould multiply request volume enormously and still add hundreds of milliseconds of latency — fails the <1s delivery target outright.
Pairwise encryption for every group member pairCost grows quadratically with group size; sender-key distribution keeps per-message group encryption linear instead.
Server-side plaintext storage for search/spam-filteringDirectly conflicts with the end-to-end encryption requirement — the server is not supposed to be able to read content at all.
Single shared connection-state table (no registry sharding)A single Redis instance can't serve lookup traffic for 150M concurrent connections — becomes an immediate bottleneck and single point of failure.
Strong consistency (synchronous cross-region replication)Would add round-trip latency to every message send; chat tolerates brief staleness in secondary regions fine — availability wins.

What breaks at 10× scale (5B DAU) and how we fix it

Problem — Connection registry becomes a hot-key bottleneck
Fix

Shard the registry further with consistent hashing across many more Redis Cluster nodes; separate read replicas for lookup-heavy traffic from write-heavy heartbeat traffic.

Problem — Gateway layer needs far more nodes for concurrent connections
Fix

Horizontally add gateway nodes behind consistent-hash-based connection placement, and tune per-node connection limits against available memory/file-descriptor limits.

Problem — Mailbox store write volume grows unsustainably
Fix

Add more Cassandra shards, tune compaction strategy for append-heavy workloads, and introduce time-bucketed partitions so old delivered messages age out of hot storage tiers.

Problem — Group fan-out for very large groups (1000+ members) creates bursts
Fix

Partition Kafka topics by group_id hash to parallelize fan-out consumption, and batch mailbox writes for members sharing the same storage shard.

Problem — Push notification providers (APNs/FCM) rate-limit at extreme volume
Fix

Batch and prioritize notifications, collapse multiple pending messages into a single "you have new messages" push rather than one push per message.

Problem — Cross-region latency as users become globally distributed
Fix

Regional gateway clusters near users, with asynchronous cross-region mailbox replication for cases where sender and recipient land in different regions — again favoring availability over strict consistency.

08 / Summary

Cheat sheet

The ten things worth remembering before walking into the room.

What

Real-time, end-to-end encrypted messaging between devices, with guaranteed offline delivery.

Core features

1:1 & group messaging · Presence · Delivery/read receipts · Media sharing.

Hardest part

Persistent connection management at 100M+ concurrent sockets.

Key NFRs

<1s delivery latency · Zero message loss · Favor Availability over Consistency (AP).

Connection strategy

Sticky Gateway connections + a distributed connection registry (Redis) for O(1) routing.

Delivery guarantee

Durable mailbox write before push attempt · monotonic per-conversation sequence numbers.

Encryption

Signal Protocol — X3DH handshake + Double Ratchet · sender keys for group fan-out.

Key databases

Sharded Cassandra (mailboxes) · Redis (connection registry + presence) · Blob+CDN (media).

Sharding keys

Mailboxes by recipient_id · Connection registry by user_id, consistent hashing.

Biggest tradeoff

Server can never read content — everything from search to spam-filtering must work on ciphertext or metadata only.
